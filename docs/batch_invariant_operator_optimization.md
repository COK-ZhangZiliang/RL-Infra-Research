# Batch-Invariant Operator Optimization for TOP

## Position

The TOP performance problem should be treated as an operator problem.

TOP requires batch-invariant execution: for a fixed request, logits and sampled
logprobs should not change when other requests are added to the same batch. The
strict fallback that guarantees this property computes each sample with a
canonical per-sample projection. That is correct, but it loses much of the
efficiency of a regular batched GEMM.

Therefore the main research target is:

> Design projection / row-linear operators that preserve batch invariance while
> recovering the tiling, reuse, and scheduling efficiency of regular batched
> GEMM.

Generic RL optimizations, such as avoiding unused vocabulary logits or fusing
sampled-logprob gather, are useful but orthogonal. They should not be counted as
the core TOP-specific optimization unless the same operator correctness contract
is preserved.

## Measured Answer

### Correctness Contracts

The benchmark now separates two different contracts:

1. **Batch-invariant TOP contract**  
   Compare each mode's mixed-batch output with the same mode's single-sample
   output:

   ```text
   op(X, W)[i] == op(X[i:i+1], W)[0]
   ```

   This is the primary TOP operator contract.

2. **Torch-single drift contract**  
   Compare a mode's output with the original `torch` single-sample output. This
   is stricter. It asks whether a replacement operator preserves the exact
   no-TOP numerical path, not only batch invariance.

These two contracts should not be mixed. A kernel can be perfectly
batch-invariant while still drifting from `torch` single-sample numerics.

### What Is The Strict-Fallback Loss?

Using `torch_batched_3d` as the no-TOP performance baseline and
`canonical_per_sample_loop` as the strict 0-drift fallback:

| setting | strict TOP overhead vs no-TOP |
|---|---:|
| all no-TF32 records, mean | `1.387x` |
| batch 8, BF16/no-TF32 | `1.601x` |
| batch 8, FP16/no-TF32 | `1.599x` |
| batch 8, FP32/no-TF32 | `1.425x` |
| batch 8, FP32+TF32 | `1.407x` |

So the measured strict 0-drift fallback costs roughly **38% on average** and
about **40-60% at batch 8**, depending on dtype and math mode.

This is the right denominator for a TOP-specific optimization claim.

### What Do Existing Batch-Invariant Kernels Recover?

The existing `batch_inv_persistent` kernel is batch-invariant across all measured
no-TF32 records:

| setting | strict 0-drift fallback | `batch_inv_persistent` | recovery |
|---|---:|---:|---:|
| batch 8, BF16/no-TF32 | `1.601x` | `1.395x` | `1.15x` faster than fallback |
| batch 8, FP16/no-TF32 | `1.599x` | `1.401x` | `1.14x` faster than fallback |
| batch 8, FP32/no-TF32 | `1.425x` | `0.848x` | faster than no-TOP baseline |
| FP16 no-TF32, mean | `1.391x` | `1.358x` | small |
| BF16 no-TF32, mean | `1.398x` | `1.341x` | small |
| FP32 no-TF32, mean | `1.371x` | `0.908x` | near-zero / faster |

The important conclusion is:

- strict 0-drift canonical fallback is expensive;
- `batch_inv_persistent` is a valid batch-invariant operator in the measured
  contract;
- for FP32/no-TF32 it is already near-zero overhead, but it drifts from
  `torch` single-sample numerics by up to about `0.0032` sampled-logprob;
- for FP16/BF16 the regular `torch_batched_3d` path is already certified
  batch-invariant in this benchmark, so the best measured TOP plan should use
  `torch`, not the slower persistent kernel.

### Where Can We Get True Near-Zero Overhead?

I implemented the experimental operator:

- `/mnt/data/RL-Infra-Research/experiments/certified_batch_invariant_projection.py`

It exposes `certified_top_projection(x, w)`, which selects the fastest
certified path for the measured contract:

- FP16/BF16/no-TF32: regular `torch_batched_3d`;
- FP32+TF32: regular `torch_batched_3d`;
- FP32/no-TF32: SGLang `batch_inv_persistent`;
- uncertified or strict 0-drift mode: canonical per-sample fallback.

The validator passes all measured contracts with overhead gate `<=1.05x`:

| dtype/math | selected mode | batch-invariant runs | mean overhead |
|---|---|---:|---:|
| BF16/no-TF32 | `certified_top_projection` | `5/5` | `1.009x` |
| FP16/no-TF32 | `certified_top_projection` | `5/5` | `1.000x` |
| FP32/no-TF32 | `certified_top_projection` | `5/5` | `0.868x` |
| FP32+TF32 | `certified_top_projection` | `5/5` | `0.992x` |

Across the no-TF32 benchmark records, `certified_top_projection` is `15/15`
batch-invariant with mean overhead `0.959x`. Including the FP32+TF32 boundary
benchmark, it is `20/20` batch-invariant under the measured contract.

This suggests a TOP-specific planner:

1. Use a certified fast path when the existing backend is empirically
   batch-invariant for the dtype/shape/math-mode contract.
2. Use a certified persistent batch-invariant kernel when the normal backend is
   not batch-invariant.
3. Use canonical fallback only when neither fast path is certified.

This is still TOP-specific because the planner is selected by the TOP
correctness contract, not by generic throughput heuristics.

## Bottleneck Analysis

### Strict Fallback Bottleneck

The strict fallback computes:

```text
for sample in batch:
    logits[sample] = hidden[sample] @ W
```

This preserves the single-sample numerical path, but it introduces four major
costs:

1. **Reduced GEMM M dimension**  
   Regular projection sees `M = batch * seq_len`; the fallback repeatedly sees
   `M = seq_len`.

2. **Poor launch and scheduler amortization**  
   A loop over samples repeatedly invokes smaller GEMMs instead of one larger
   scheduled kernel.

3. **Lower occupancy for medium shapes**  
   One sample may not create enough output tiles to keep all SMs busy.

4. **Weaker weight reuse**  
   The same `W` tiles are revisited per sample instead of being reused across a
   larger row set.

### Existing Persistent Kernel Bottleneck

`batch_inv_persistent` satisfies the measured batch-invariance contract, but the
current measurements show two different gaps:

- it is still about `1.39-1.40x` at batch 8 for FP16/BF16, so it should not
  replace a certified `torch` fast path for those dtypes;
- for FP32/no-TF32 it is batch-invariant and fast, but it drifts from
  `torch` single-sample numerics because its arithmetic path is different;
- `tp_inv` and `tp_inv_optim` satisfy batch invariance in this projection
  harness, but they are much slower than the alternatives;
- `tp_inv_optim` hits shared-memory resource errors for FP16/BF16 on RTX 3090.

Likely kernel-level causes:

- tile sizes are not autotuned for this projection shape and SM86 shared-memory
  budget;
- the reduction/accumulation path is canonical for itself, but not identical to
  the original `torch` single-sample path;
- persistent scheduling exists, but not enough of the batched GEMM locality is
  recovered for FP16/BF16;
- TP-invariant kernels solve a related but different contract and cannot be
  assumed to be the fastest projection operator.

## Implemented Operator: Certified TOP Projection

The implemented experimental solution is a certified planner wrapped as a
projection operator, plus an optional stricter fallback for deployments that
require exact `torch` single-sample numerics.

### Level 1: Certified Fast Path Inside `certified_top_projection`

Before selecting a slower TOP kernel, certify whether the normal backend already
satisfies:

```text
op(X, W)[i] == op(X[i:i+1], W)[0]
```

for the concrete dtype, math mode, shape family, GPU architecture, and backend
version.

If the certificate passes, use `torch_batched_3d` or `torch_flattened_2d`
directly. This gives true near-zero overhead because the TOP path uses the same
operator as no-TOP.

The current evidence supports this for the measured FP16, BF16, and FP32+TF32
projection configurations.

### Level 2: Certified Persistent Batch-Invariant Path Inside `certified_top_projection`

When the regular backend is not batch-invariant, use the fastest certified
persistent kernel. In the current benchmark, this is `batch_inv_persistent` for
FP32/no-TF32:

```text
FP32/no-TF32:
  torch_batched_3d        -> not batch-invariant, 1/5 runs
  certified_top_projection -> batch-invariant, 5/5 runs, 0.868x mean overhead
```

This already satisfies the primary TOP operator contract in the measured
configuration.

### Level 3: `canonical_batched_persistent_v2` For 0-Drift Mode

If the deployment also requires exact agreement with the original
`torch` single-sample numerics, use a new canonical batched persistent GEMM
instead of `batch_inv_persistent`.

Contract:

```text
X: [B, S, H]
W: [H, V]
C: [B, S, V]

canonical_batched_persistent_v2(X, W)[i]
  == canonical_batched_persistent_v2(X[i:i+1], W)[0]
```

Optional stricter contract:

```text
canonical_batched_persistent_v2(X, W)[i]
  == torch_matmul(X[i:i+1], W)[0]
```

The optional stricter contract is the harder one. It may require matching
cuBLAS / PyTorch's single-sample FP32 accumulation path, not merely being
internally batch-invariant.

Key design:

1. **Flatten rows, not semantics**  
   Compute over `M = B * S`, but ensure each output row follows the same
   numerical path regardless of batch composition.

2. **Fixed K-block order and reduction tree**  
   K tiles are accumulated in a deterministic order. Any split-K or parallel
   reduction must merge partial sums through a fixed tree that does not depend
   on batch size.

3. **Grouped M/N tile ordering**  
   Schedule neighboring row tiles with neighboring vocabulary tiles to improve
   L2 reuse of `W`, following the same locality principle as high-performance
   Triton GEMM.

4. **Persistent scheduling**  
   Persistent CTAs pull canonical output tiles from a deterministic logical tile
   stream. Physical SM assignment may vary, but the arithmetic path of each
   output element must not.

5. **Shape-specialized autotuning**  
   Tune `BLOCK_M`, `BLOCK_N`, `BLOCK_K`, `num_warps`, `num_stages`, and
   `GROUP_M` for TOP projection shapes. The RTX 3090 result specifically shows
   that shared-memory budgets must be part of the autotune constraint.

6. **Separate optional RL fusion**  
   A sampled-logprob-only fused path may be useful for PPO old-logprob
   recomputation, but it is a separate optimization. The core TOP operator must
   first be validated for full projection semantics.

## Validation Gates

For the measured contracts, `certified_top_projection` passes the primary TOP
gates:

| gate | requirement |
|---|---|
| G1 batch-invariance | passed: bitwise equality between same-mode single-sample and mixed-batch sampled logprobs |
| G2 overhead | passed: `<= 1.05x` versus `torch_batched_3d` on the same shape |
| G3 batch coverage | passed: batch sizes `1, 2, 4, 8, 16` |
| G4 dtype coverage | passed: FP16, BF16, FP32/no-TF32, FP32+TF32 in this benchmark |
| G5 certification | passed: planner selects `certified_top_projection` for every measured contract |
| G6 fallback | partially implemented: uncertified dtypes fall back to canonical loop; broader shape certificates are future work |
| G7 optional 0-drift | available via `certified_top_projection_0drift`, but costs about `1.37x`; near-zero 0-drift remains open |

## Main Claim After This Pivot

The defensible claim is:

> A strict 0-drift TOP fallback pays about `1.38x` average projection latency in
> the measured benchmark because it sacrifices batched GEMM efficiency. Under
> the primary batch-invariant TOP contract, the implemented
> `certified_top_projection` operator recovers near-zero overhead for all
> measured dtype/math-mode contracts: FP16/BF16 and FP32+TF32 use the certified
> regular torch path, while FP32/no-TF32 uses `batch_inv_persistent`. The
> remaining open problem is the stricter 0-drift variant: matching original
> `torch` single-sample numerics while keeping near-zero overhead.
